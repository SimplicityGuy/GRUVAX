# Phase quick-260608-f14 Plan 01: Fix react-qr-code Type Declaration CI Failure Summary

**One-liner:** Type shim + tsconfig `paths` redirect for react-qr-code@2.1.1 whose bundler-mode exports map lacks a `types` condition, restoring the CI tsc gate to green.

## What Was Built

### Task 1: Add ambient declaration shim for react-qr-code

Created `frontend/src/react-qr-code.d.ts` as a module-style TypeScript declaration file (no `declare module` wrapper) and added a `paths` entry in `frontend/tsconfig.app.json` redirecting `react-qr-code` imports to this shim.

**CI gate result:**

```
$ npm --prefix frontend exec tsc -- --noEmit -p frontend/tsconfig.app.json; echo "exit=$?"
exit=0
```

No errors. The pre-commit tsc hook also passed cleanly on commit.

## Root Cause (deeper than the plan anticipated)

The plan correctly identified the primary cause: `react-qr-code@2.1.1`'s `exports` map has no `"types"` condition, so under `moduleResolution: bundler` TypeScript ignores the shipped `types/index.d.ts`.

The actual error, however, was TS2786 (not TS7016) because TypeScript fell through to a **globally installed** `react-qr-code@2.0.21` at `~/node_modules/react-qr-code/types/index.d.ts`. That file's `class QRCode extends React.Component<QRCodeProps, any>` declaration fails the React 19 / TypeScript 6.0.3 JSX class constructor check (`Component<any, any, any>` requires `context`, `setState`, `forceUpdate`, `props`, `state` to be visible).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] tsconfig `paths` required; ambient shim alone is not sufficient**

- **Found during:** Task 1 (after first shim creation attempt)
- **Issue:** An ambient `declare module 'react-qr-code'` in a user `.d.ts` file is treated by TypeScript as module AUGMENTATION (merged with, not replacing, the existing resolution). The plan assumed the shim would act as the primary declaration, but TypeScript's resolution already found the global `~/node_modules/react-qr-code@2.0.21` types, making the shim a merge target.
- **Fix:** Changed the shim to a module-style `.d.ts` (direct exports, no `declare module` wrapper) and added `"paths": { "react-qr-code": ["./src/react-qr-code"] }` to `frontend/tsconfig.app.json`, redirecting module resolution to the shim before the fallback walk reaches the global node_modules.
- **Files modified:** `frontend/tsconfig.app.json`
- **Commit:** 14d6639

**2. [Rule 1 - Bug] Use `React.ComponentType` not class-extends declaration**

- **Found during:** Initial diagnosis of TS2786
- **Issue:** The shipped `class QRCode extends React.Component<QRCodeProps, any>` declaration (from the global 2.0.21 types) does not satisfy TypeScript 6.0.3 + React 19's JSX class constructor check. The check requires `QRCode` instance to explicitly expose `context`, `setState`, `forceUpdate`, `props`, `state` — which class inheritance from an ambient module doesn't guarantee structurally.
- **Fix:** Declared the default export as `React.ComponentType<QRCodeProps>` (union of functional and class component types), which is unambiguously valid for JSX in React 19.
- **Files modified:** `frontend/src/react-qr-code.d.ts`
- **Commit:** 14d6639
