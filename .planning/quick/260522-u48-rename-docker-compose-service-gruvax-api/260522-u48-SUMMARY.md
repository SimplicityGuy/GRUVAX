# Quick Task 260522-u48 Summary

Renamed the Compose service key `gruvax-api` → `api` so the container is now
`gruvax-api-1` (project `gruvax` + service `api`) instead of the doubled
`gruvax-gruvax-api-1`.

## Changes

- **compose.yaml** — service key `gruvax-api:` → `api:`; updated the service-name
  references in the header comments (stack title, DB-connectivity notes, runbook).
  Left `image: gruvax-api:local` and `MQTT_USERNAME: ${MQTT_USERNAME:-gruvax-api}`
  unchanged (neither is the service name).
- **README.md** — updated the two service references (DB-connectivity note +
  repo-layout compose description) to `api`.

## Verification

- `docker compose config --services` → `gruvax-dev-pg`, `mosquitto`, `api`.
- Recreating the stack required `down`/`up` (the first `up` left the new container
  on no network because the old `gruvax-gruvax-api-1` still held :8000; a clean
  `docker compose down && docker compose up -d` — volumes preserved — fixed it).
- `docker compose ps` → container `gruvax-api-1`, port `0.0.0.0:8000->8000`,
  on `gruvax_internal`.
- `curl localhost:8000/api/health` → 200 (`db: ok`, `mqtt: ok`).
- `docker compose exec api …` works; `docker compose exec gruvax-api …` now
  correctly reports "service gruvax-api is not running".

## Notes

- Operators with an existing stack must run `docker compose up -d --remove-orphans`
  (or `down`/`up`) once to drop the old `gruvax-gruvax-api-1` orphan container.
