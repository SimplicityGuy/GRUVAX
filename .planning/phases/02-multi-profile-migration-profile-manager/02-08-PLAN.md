<objective>
Close UAT Gap (test 8, major): the admin profile drawer is stuck on "Syncing… 3,000 items processed" and never transitions to CONNECTED + fires the completion toast without a manual page refresh.

The backend is correct: `GET /api/admin/profiles/{id}` returns `last_sync_status:"ok"`, `status:"connected"`, `last_sync_item_count:3000` after the sync completes. The bug is the frontend poll in `ProfileDrawer.tsx`:

  refetchInterval: (query) => query.state.data?.last_sync_status === 'in_progress' ? 2000 : false

This stops polling on ANY non-`'in_progress'` value. Because the backend's `in_progress` → terminal transition is observed across separate writes (the early `in_progress` write in `trigger_sync` / `_record_failure`, vs. the terminal `ok` write inside `_swap_inside_tx`), a poll tick can fetch a row whose `last_sync_status` is momentarily `null`/transient, halt polling, and never observe the terminal `'ok'`. The terminal-state `useEffect` (status==='ok' → `setConnectState('idle')` + toast) then never fires.

Purpose: Make the poll robust — keep polling UNTIL a terminal status is observed — and confirm the backend writes the terminal status + item_count atomically so the poll always converges on `'ok'`.
Output: A poll-until-terminal `refetchInterval`, a regression vitest test that reproduces the transient-tick bug and asserts the drawer transitions + toast fires, and a documented/verified-atomic backend completion write.
</objective>
