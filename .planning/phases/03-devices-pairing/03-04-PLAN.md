<interfaces>
Backend endpoints (from 03-02 / 03-03, now live):
- POST /api/devices/pairing-codes -> {code: string, expires_at: string}
- GET /api/devices/me -> {state: "unpaired"|"pending"|"paired"|"revoked", profile_id?: string}
- POST /api/admin/devices/bind {code, profile_id?, display_name?} -> 200 | 404 {type:"code_not_found"|"code_expired"} | 429 {type:"rate_limited"}
- GET /api/admin/devices -> {paired: DeviceRow[], pending: DeviceRow[], revoked: DeviceRow[]}
- PATCH /api/admin/devices/{id} {display_name?|profile_id?(null to unbind)}
- POST /api/admin/devices/{id}/revoke ; POST /api/admin/devices/{id}/reinstate ; DELETE /api/admin/devices/{id}
- GET /api/session -> now also returns device_id?: string | null, is_device_paired?: boolean

DeviceRow shape: { id, display_name, state, profile_id|null, profile_name?, last_seen_at|null }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| kiosk SPA → /api/devices/* | renders server-issued code; never reads the HttpOnly fingerprint |
| admin SPA → /api/admin/devices/* | PIN-gated mutations behind PinOverlay |
| SPA lifecycle filtering | client filters SSE events by device_id (non-secret) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-15 | Information Disclosure | fingerprint read by client JS | mitigate | fingerprint cookie is HttpOnly (server-set in 03-01); SPA only ever sees device_id; PairView never requests the fingerprint |
| T-03-16 | Spoofing | unauthenticated access to admin device UI | mitigate | /admin/devices behind the existing PinOverlay flow; all mutations call PIN-gated endpoints (touchpoint #8) |
| T-03-17 | Tampering | client bypasses revoke by ignoring SSE | accept | UI is convenience only; backend per-request guard (03-03) is authoritative — a revoked kiosk gets 403 on its next request regardless of SPA behavior |
</threat_model>
