## Surface 1 — `/pair` Route (Kiosk, Full-Screen)

### Layout

Full-viewport flex column, vertically and horizontally centered. No scrolling — all content fits within `100dvh`. This is a wall-mounted display; legibility from 1–2 meters is the design constraint.

```
┌─────────────────────────────────────┐
│                                     │  top: 48px padding
│         [GRUVAX icon mark]          │  icon mark at 48px, --gruvax-blue
│                                     │  gap: 32px
│       PAIR THIS SCREEN              │  Barlow Condensed 900, 48px, --gruvax-blue
│                                     │  gap: 16px
│   Enter this code in the admin app  │  Space Grotesk 400, 18px, --gruvax-text-secondary
│                                     │  gap: 48px
│   ┌───────────────────────────┐     │
│   │         4  8  2  7        │     │  Code card: DM Mono 96px, --gruvax-blue
│   │                           │     │  Card: --gruvax-off-white, --gruvax-radius-xl
│   └───────────────────────────┘     │  Card padding: 24px 48px
│                                     │  gap: 24px
│         ● WAITING FOR PAIRING…      │  Spinner + Barlow Condensed 700, 16px
│              4:59 remaining         │  DM Mono 500, 16px, --gruvax-text-muted
│                                     │
└─────────────────────────────────────┘
```

### States

| State | Visual |
|-------|--------|
| Loading (fetching code from server) | Code card shows 4 placeholder dashes "— — — —" in DM Mono 96px muted; no countdown visible |
| Active (code displayed, countdown running) | Code visible; countdown counting down; "WAITING FOR PAIRING…" spinner |
| Countdown ≤ 60s | Countdown text transitions to `--gruvax-warning`; no other change |
| Code expired (countdown reaches 0:00) | Code card pulses once with `--gruvax-blue-light` flash (150ms), then blanks and shows "Generating new code…" in `--gruvax-text-muted`; new code auto-fetched immediately |
| Pairing successful (poll returns `paired`) | Code card transitions: border springs to `--gruvax-yellow` (300ms spring), text changes to "PAIRED — navigating…"; after 800ms auto-navigate to bound-profile search UI; no manual user action required |
| Already paired (D3-03 routing rule) | Route immediately redirects to bound-profile search UI — `/pair` page never renders |

### Countdown behavior

- Timer starts at 5:00 (300 seconds) on code receipt.
- Client-side `setInterval` countdown; re-request on reaching 0:00.
- Format: `M:SS` (e.g., "4:59", "0:12"). Use DM Mono to prevent layout shift.
- Server clock is authoritative — compute remaining time from `expires_at` in the API response minus `Date.now()`, not from a client-side epoch.

### Polling

- `GET /api/devices/me` every 3 seconds while `/pair` is mounted.
- Use TanStack Query `refetchInterval: 3000` while `state !== 'paired'`.
- On `state === 'paired'`: stop polling, run success transition, navigate.

### "PAIR THIS SCREEN" affordance (D3-02)

Place on two existing surfaces:

**On `/select` ProfilePicker:**
- Add a secondary action below the profile card list.
- Button: full-width, `--gruvax-yellow` background, `--gruvax-blue-darker` text, Barlow Condensed 900 18px ALL CAPS, min-height 48px, `--gruvax-radius-md`.
- Label: "PAIR THIS SCREEN AS A DEVICE"
- Position: below the profile list, above any footer. Separated by `--gruvax-space-5` (24px) from the list.

**On onboarding screen (0-profile state):**
- Add below the primary "Add a profile" call-to-action.
- Same button style as above.
- Label: "PAIR THIS SCREEN AS A DEVICE"
- Instruction below button: "Already have profiles set up? Link this screen to one." — Space Grotesk 400, 14px, `--gruvax-text-muted`.

*Source: CONTEXT.md D3-02; ROADMAP success criterion #1*

**Last-seen formatting:**
- ≤ 2 minutes: "just now"
- 3–59 minutes: "Nm ago"
- 1–23 hours: "Nh ago"
- ≥ 24 hours: "Nd ago"
- Never seen: "never"

## Device Drawer (Bottom Sheet)

Reuses the `record-picker-sheet` / `sheet-*` CSS classes from `admin.css` (established in v1.0 Phase 5, reused in P2 ProfileDrawer — D2-11). No new sheet CSS.

### Structure

```
[sheet-scrim]
  [record-picker-sheet]
    [sheet-drag-pill]
    [sheet-body]
      [sheet-heading]   — Device display name, ALL CAPS
      [device-id-line]  — DM Mono 14px: "ID: a3f7c2d1b2c3d4e5"
      [sheet-error]     — (conditional) error message
      [sheet-actions]
        — Context-sensitive action buttons (see below)
      [sheet-cancel-btn] — "CANCEL"
```

### Actions by device state

#### New device bind (ADD DEVICE flow — "Enter code")

1. "ENTER PAIRING CODE" sheet heading.
2. Code display: 4 DM Mono character boxes (each showing digit or "·" placeholder), 36px, `--gruvax-blue`, centered. Updates live as digits entered via NumericKeypad.
3. `NumericKeypad` component (reused verbatim, no changes). Limited to 4 digits; auto-submits on 4th digit.
4. Submit button: "BIND DEVICE" — disabled until 4 digits entered; blue primary style.
5. On success: drawer transitions to "BOUND" confirmation state showing device name, then closes with a toast.
6. On error "code_not_found": inline `.sheet-error` "That code wasn't found. Check the kiosk screen and try again."
7. On error "code_expired": inline `.sheet-error` "That code has expired. Ask the kiosk to generate a new one."
8. On error "rate_limited": inline `.sheet-error` "Too many attempts. Wait a moment and try again."

#### PENDING device drawer

Actions (in order):
1. "BIND TO PROFILE" — opens profile-picker bottom sheet (reuse or adapt the existing profile list pattern); on pick: auto-submits bind with the last pending code if available, else prompts for code entry.
2. "RENAME" — inline text input (same pattern as ProfileDrawer rename mode).
3. "REVOKE" — destructive, see destructive section below.

#### PAIRED device drawer

Actions (in order):
1. "RENAME" — inline text input, same pattern as ProfileDrawer rename mode.
2. "CHANGE PROFILE" — opens profile-picker; on pick: PATCH `profile_id` on the device row.
3. "UNBIND" — removes `profile_id` (device becomes PENDING); confirmation: "Remove the profile binding? The kiosk will return to the profile picker." Primary/Cancel — no separate confirm screen (non-destructive).
4. "REVOKE DEVICE" — destructive, see destructive section below.

#### REVOKED device drawer

Actions (in order):
1. "REINSTATE" — clears `revoked_at`, returns device to PENDING state; no confirmation needed (reversal action).
2. "DELETE PERMANENTLY" — hard-delete; see destructive section below.

## Destructive Action Patterns

### REVOKE DEVICE

Inline confirmation within the drawer (no separate screen — matches P2 ProfileDrawer delete-confirm mode):

1. Tap "REVOKE DEVICE" button: action section replaces with a confirmation block.
2. Confirmation heading: "Revoke this device?" — Space Grotesk 500, 16px, `--gruvax-error`.
3. Confirmation body: "The kiosk will be locked out immediately. It can be reinstated later." — Space Grotesk 400, 14px, `--gruvax-text-secondary`.
4. Two buttons stacked:
   - "REVOKE" — full-width, `--gruvax-error` background, white text, Barlow Condensed 900 16px.
   - "CANCEL" — ghost, `--gruvax-text-muted`, Barlow Condensed 700 16px.
5. On confirm: optimistic update → badge flips to REVOKED; SSE will push the live update to the kiosk.

### DELETE PERMANENTLY (revoked device)

Same inline confirmation pattern:

1. Confirmation heading: "Delete this device record?"
2. Confirmation body: "This removes the device history. The kiosk will need to pair again from scratch."
3. Buttons: "DELETE" (destructive) / "CANCEL".

### UNBIND (non-destructive)

Inline single confirmation prompt only — no separate confirm screen:

1. "Remove the profile binding?" — body text.
2. "UNBIND" (primary, `--gruvax-blue`) / "CANCEL".

*Source: P2 ProfileDrawer delete-confirm pattern; CONTEXT.md D3-02; REQUIREMENTS DEV-02*

## Accessibility

| Constraint | Implementation |
|-----------|----------------|
| Blue-on-white body text | `--gruvax-blue` on `--gruvax-white` = 7.2:1 (AAA) — safe for all text sizes |
| Yellow-on-blue buttons | Only on ≥ 18px Barlow Condensed 900 elements; contrast ~3.1:1 — large text only |
| Yellow-on-blue badges | NOT used — badges use `--gruvax-success` / `--gruvax-warning` / `--gruvax-error` on faint tint backgrounds |
| Kiosk code digits | `--gruvax-blue` on `--gruvax-off-white` → 7.2:1 (AAA) — no accessibility exception needed at 96px |
| Countdown warning color | `--gruvax-warning` (#E6A800) on white = 3.1:1 — use only at ≥ 16px DM Mono 500 |
| Touch targets (kiosk) | Minimum 64×64px on `/pair`; no small tap targets on a wall-mounted screen |
| Touch targets (mobile admin) | Minimum 48px height on all interactive elements (device cards, drawer actions, keypad keys) |
| Focus rings | `outline: 2px solid var(--gruvax-blue); outline-offset: 2px` (keyboard navigation; yellow for icon buttons on blue backgrounds) |
| Drawer focus trap | Same focus-trap-react pattern as `ProfileDrawer` (via `sheetRef`) |
| ARIA — code display | `role="status"` + `aria-live="polite"` on the code display; `aria-label="Pairing code: {code}"` |
| ARIA — countdown | `aria-live="off"` (avoid announcing every second); announce at 60s, 30s, 10s milestones via `aria-live="assertive"` on a hidden announcer node |
| ARIA — device list | `<ul>` with `aria-label="Device list"` per group; each `<li>` wraps the card button |
