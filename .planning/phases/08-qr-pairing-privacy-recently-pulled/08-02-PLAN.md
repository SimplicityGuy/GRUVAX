<objective>
DEV-04: add a QR pairing path alongside the existing 4-digit PIN on the kiosk pairing
screen. The QR encodes a bind URL carrying the current pairing code (D-01/L-02 — the code
is the opaque short-TTL token, never a credential). Scanning lands on the existing
PIN-gated admin bind page (D-02) prefilled with the code; the admin completes pairing with
an explicit one-tap confirm (D-04). Both pairing paths funnel through the existing
`handleBind` → `POST /api/admin/devices/bind` call site, so audit entries are identical
(L-03, success criterion 1).

Purpose: Frictionless phone-based pairing with no new security model and no second code path.

Output: react-qr-code installed; QR block in PairView; prefill-confirm flow in
DeviceDrawer/DevicesManager.

## Key Decision (Pitfall 39 / L-01): QR over HTTP on the home LAN

The QR bind URL uses HTTP (not HTTPS) and encodes only the existing short-TTL single-use
4-digit pairing code, never a credential. The admin PIN gate remains the sole security
control for all binds, regardless of transport. The pairing code is already single-use
(atomic `UPDATE ... WHERE consumed_at IS NULL`) with a 5-minute TTL. This is proportionate
to the home-LAN threat model with owner-controlled physical access. No separate
faster-rotating nonce is introduced (supersedes the "60s nonce" open-decision note).
Revisit only if GRUVAX is ever exposed beyond the home LAN.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| kiosk display → phone camera | QR is publicly visible on the kiosk; whatever it encodes is readable by anyone in the room |
| phone → /admin/devices (PIN gate) | Untrusted scan target; the PIN gate is the access-control boundary |
| phone → POST /api/admin/devices/bind | Bind request crosses into privileged device-management |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-QR-01 | Information Disclosure | QR payload | mitigate | D-01/L-02: QR encodes only the short-TTL 4-digit pairing code in a bind URL, never a PIN or PAT |
| T-08-QR-02 | Elevation of Privilege | replayed/old QR (stale code) | mitigate | Existing atomic single-use consume (`UPDATE ... WHERE consumed_at IS NULL AND expires_at > NOW()`) + 5-min TTL; a stale code yields `mapBindError` |
| T-08-QR-03 | Elevation of Privilege | auto-bind on scan without confirm | mitigate | D-04: explicit one-tap "PAIR THIS DEVICE" confirm; no auto-submit from prefill |
| T-08-QR-04 | Spoofing / EoP | scanned QR binding without admin auth | mitigate | D-02: scan lands on the PIN-gated /admin/devices; PinOverlay modal requires PIN before the drawer is reachable |
| T-08-QR-05 | Repudiation | divergent audit between scan vs typed path | mitigate | L-03: both paths call the single `handleBind` → POST /api/admin/devices/bind site; identical audit entries |
| T-08-QR-SC | Tampering | npm install react-qr-code | mitigate | Pinned 2.0.21; slopcheck [OK], MIT, no postinstall (Package Legitimacy Audit in 08-RESEARCH.md) |
</threat_model>
