<objective>
Close B-02 on the backend: make `profile_id` an optional query param on `/api/search` and
`/api/locate`, falling back to the cookie-authoritative `resolve_profile_from_request` when it is
omitted, while preserving the D2-04 400-unbound / 403-mismatch validation exactly when a
`profile_id` IS supplied.

Purpose: A user who types before session bootstrap resolves the bound profile (so the frontend
omits `profile_id`) currently gets a 422 results-list error. The backend must instead resolve the
authoritative profile from the `gruvax_browse_binding` cookie and return 200 scoped to that profile.

Output: RED integration tests (written first) + the tolerant handlers in `search.py` and
`locate.py`. Restores API-02 end-to-end (search/locate off the per-profile cache for the
omitted-param path).
</objective>
