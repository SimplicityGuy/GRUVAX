<objective>
Fix the second `just compose-smoke` blocker surfaced by `/gsd-verify-work 1` after the 01-09 fix landed. With the duplicate-image-tag bug resolved in 01-09, `docker compose up` now reaches the api container's entrypoint — at which point `alembic upgrade head` crashes during migration **0002** with `relation "collection_items" does not exist`.

**Root cause:** migration 0002 creates `gruvax.v_collection` whose SELECT body references **unqualified** `collection_items`, `releases`, `artists`. In dev/CI those resolve via `search_path` to `gruvax_dev.*`; in production they resolve to `discogsography.*`. On a fresh `postgres:18` compose volume **neither schema exists**, so 0002 errors and the container crash-loops before migration **0009** gets a chance to DROP the view per D-19.

**The fix (Option A, smallest possible scope):** insert a `gruvax_dev` bootstrap step in `docker-entrypoint.sh` BEFORE `alembic upgrade head`. The bootstrap uses idempotent `CREATE SCHEMA IF NOT EXISTS gruvax_dev` + `CREATE TABLE IF NOT EXISTS` for the three minimum tables 0002's view body references (`collection_items(id, release_id, updated_at)`, `releases(id, title, label, catalog_number, format, year, fts_vector, primary_artist_id)`, `artists(id, name)`). The tables are STUBS — empty placeholders that let 0002 create an empty view, after which 0009 drops it during the same `alembic upgrade head` run and the chain completes cleanly.

**Critical guards:**
- **Environment guard:** wrap the bootstrap in `if [ "${GRUVAX_ENV:-production}" = "development" ]` — matches the existing guard on the synth seed at the bottom of the same script. Production deployments target the real `discogsography` schema per D-12 and the bootstrap must never run there.
- **Idempotency:** `CREATE SCHEMA IF NOT EXISTS` + `CREATE TABLE IF NOT EXISTS` make the bootstrap a no-op when the dev DB already has real discogsography-mock tables (the documented local-dev state per `integration_test_harness` memory: "shared dev Postgres … gruvax_dev populated by hand-managed seed scripts").
- **Stub-only documentation:** add an inline comment block above the bootstrap explicitly noting these are placeholder stubs to let migration 0002 succeed — never to be seeded with real data — so future code doesn't try to populate them.

Purpose: Unblocks compose-smoke's alembic-upgrade step. With this and 01-09 in place, `just compose-smoke` reaches the gruvax-api healthy state and downstream UAT tests 2-5 can finally execute.

```sh
#!/bin/sh
# GRUVAX container entrypoint script.
# Uses 'python -m <module>' invocations to avoid hardcoded shebang paths
# that arise when the venv is built in one directory and copied to another.
set -e

PYTHON="/app/.venv/bin/python"

# Wait for Postgres to accept connections before migrating — a cold
# `docker compose up` starts the DB and the API concurrently, so without this
# `alembic upgrade head` would fail and crash-loop the container (WR-04).
echo "Waiting for database to become available..."
i=0
until "$PYTHON" - <<'PY' 2>/dev/null
import os, psycopg
dsn = os.environ["DATABASE_URL"].replace("postgresql+psycopg://", "postgresql://")
psycopg.connect(dsn, connect_timeout=3).close()
PY
do
    i=$((i + 1))
    if [ "$i" -ge 30 ]; then
        echo "Database not reachable after 30 attempts — aborting." >&2
        exit 1
    fi
    sleep 2
done
echo "Database is up."

# Run Alembic migrations
"$PYTHON" -m alembic upgrade head
```
