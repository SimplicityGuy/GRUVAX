# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase introduces **no new screens**. All surfaces are extensions of existing Phase 2/3/8
components. The contract covers: five surfaces, all polished extensions.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — plain CSS + GRUVAX design token system |
| Preset | not applicable |
| Component library | none (hand-authored DOM components) |
| Icon library | inline SVG (Lucide-pattern, `stroke="currentColor"`) |
| Font | Barlow Condensed 700 · Space Grotesk 400/700 · DM Mono 400 (Google Fonts) |

Source of truth: `design/gruvax-design-tokens.css` + `design/gruvax-design-language.md`.
Enforce: no hardcoded hex values anywhere in P4 CSS or TSX — consume tokens only.

## Spacing Scale

Declared values (consumed from `--gruvax-space-*` tokens — never hardcode):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`--gruvax-space-1`) | Icon-to-label gap, inline indicator gaps |
| sm | 8px (`--gruvax-space-2`) | Badge padding, spinner-to-label gap |
| md-sm | 12px (`--gruvax-space-3`) | Kiosk banner vertical padding (matches `staleness-bar`) |
| md | 16px (`--gruvax-space-4`) | Card content padding, row padding |
| lg | 24px (`--gruvax-space-5`) | Section vertical rhythm (between cards) |
| xl | 32px (`--gruvax-space-6`) | Section break (PROFILES heading to first card) |
| 2xl | 48px (`--gruvax-space-7`) | Page-level top/bottom padding |
| 3xl | 64px (`--gruvax-space-8`) | Reserved — page layout only |

Exceptions:
- Touch targets: minimum `44px` height on all interactive elements (buttons, cadence select, sync-now button) — not a spacing value, it is a minimum tap-target floor.
- Diagnostics cards grid: use CSS Grid with `gap: var(--gruvax-space-5)` (24px), not larger.

## Typography

Three-font system. P4 introduces no new type sizes — all values already declared in the design tokens.

| Role | Font | Size | Weight | Line Height | Usage in P4 |
|------|------|------|--------|-------------|-------------|
| Section heading | Barlow Condensed | 24px (`--gruvax-text-display-md`) | 700 | 1.1 (`--gruvax-leading-tight`) | "PROFILES" heading on `/admin/diagnostics` |
| Row label / badge text | Barlow Condensed | 16px (`--gruvax-text-display-sm`) | 700 | 1.1 | Profile name in diagnostics card; status badge copy; cadence select label |
| Body / instructions | Space Grotesk | 14px (`--gruvax-text-body-sm`) | 400 | 1.5 (`--gruvax-leading-normal`) | Card metadata values, cadence sub-label, banner copy (≥18px exception below), error/empty text |
| Data readout | DM Mono | 14px (`--gruvax-text-mono`) | 400 | 1.5 | `last_sync_at`, `last_sync_item_count`, elapsed sync seconds |
| Kiosk banner (accessibility floor) | Space Grotesk | 18px (`--gruvax-text-body-lg`) | 400 | 1.5 | Re-auth inline banner — yellow-on-blue-darker at 18px meets WCAG large-text |

Type scale is exactly four sizes: **14px · 16px · 18px · 24px**. No 5th size (no 12px) appears anywhere in P4 — sub-labels use 14px muted, not 12px.

Font weights in use: **400** (body/data) + **700** (display/labels/headings). Exactly two weights — never introduce a third for P4 work.

> Note: P4 demotes the diagnostics section heading from the design language's default 900 to **700**. Barlow Condensed 700 at 24px still reads as a strong display heading and keeps P4 within the two-weight cap. P4 does NOT reuse the existing `.diag-heading` class (which is 900); it declares a 700-weight heading class instead.

## Color

All via `var(--gruvax-*)` tokens. No hardcoded hex.

| Role | Token | Hex | Usage |
|------|-------|-----|-------|
| Dominant (60%) | `--gruvax-white` | #FFFFFF | Page background, card white surfaces |
| Secondary (30%) | `--gruvax-off-white` | #F7F9FC | Diagnostics card backgrounds (`settings-section`) |
| Accent (10%) | `--gruvax-yellow` | #FFDA00 | Re-auth kiosk banner background ONLY |
| Destructive | `--gruvax-error` | #C0392B | (No new destructive actions in P4 — reserved pattern only) |

### Semantic colors in P4

| Signal | Token | Used on |
|--------|-------|---------|
| Success (sync ok) | `--gruvax-success` (#1A7A4A) | Diagnostics card "OK" badge background tint; sync-complete toast background |
| Warning (stale) | `--gruvax-warning` (#E6A800) | Diagnostics card "STALE" badge text |
| Error (re-auth, outdated) | `--gruvax-error` (#C0392B) | Re-auth badge `profile-status-badge--re-auth-required`; diagnostics "OUTDATED" badge text |
| Blue family (structure) | `--gruvax-blue` / `--gruvax-blue-faint` | Sync spinner ring, card headings, row labels |

### Accent reserved-for list

`--gruvax-yellow` is used ONLY for:
1. The kiosk re-auth inline banner background (matching existing `staleness-bar` pattern)
2. The `sync-progress-spinner` ring border (existing `SyncProgressSection` pattern — `--gruvax-yellow` ring on `--gruvax-blue-faint` track)

Do NOT use yellow on profile diagnostics cards, cadence selector, or toast. Those use blue/success/error semantics.

## Surfaces

P4 touches five surfaces. Each surface extends existing components; no full rewrites.

### Surface 1 — `/admin/diagnostics`: PROFILES section (new section appended to existing page)

**What it is:** A new `<section class="settings-section">` appended below the existing "SYSTEM" section. Contains per-profile diagnostic cards — one card per non-deleted profile.

**Focal point:** the `ProfileStatusBadge` in each card header row — the first thing an admin scans to assess sync health. Card layout, color, and spacing all serve to make the badge the immediate read; metadata rows are secondary detail.

**Per-profile card layout:**
- Card container: `settings-section` (existing class, `--gruvax-off-white` background, `--gruvax-shadow-sm`, `--gruvax-radius-lg` border-radius, `--gruvax-space-4` padding)
- Card header row: profile display name (Barlow Condensed 16px 700 `--gruvax-blue`) + `ProfileStatusBadge` (existing component, status derived from `app_token_revoked` + `last_sync_status`)
- Metadata grid: four data rows, each in `diag-status-row` layout (label left, value right)
  - LAST SYNC: DM Mono 14px, formatted as relative time (e.g., "3h ago", "2d ago", "—")
  - STATUS: Barlow Condensed 16px badge — "OK" / "STALE" / "OUTDATED" using `.diag-badge--{ok|stale|outdated}` (existing classes)
  - ITEMS: DM Mono 14px, formatted as `{N,###}` (comma-separated integer, or "—")
  - LAST ERROR: Space Grotesk 14px `--gruvax-text-muted`, truncated to 60 chars with "…" (or "none" when null)

**Polling:** `refetchInterval: 30_000` (30 seconds) via TanStack Query on `GET /api/admin/diagnostics`. No SSE.

**Loading state:** reuse `.diag-skeleton` shimmer for each card placeholder (existing animation).

**Empty state:** "No profiles yet. Create a profile to see sync diagnostics." (Space Grotesk 14px muted, centered, `--gruvax-space-5` vertical padding).

---

### Surface 2 — Profile list / ProfileDrawer: re-auth badge (existing `ProfileStatusBadge`)

**What it is:** The `ProfileStatusBadge` component already has a `re-auth-required` status. P4 wires `app_token_revoked: true` from the API response to render this badge in the `ProfileCard` and `ProfileDrawer`.

**No new CSS needed.** The badge class `profile-status-badge--re-auth-required` is already defined (error-tinted background, `--gruvax-error` text). P4 is purely a data-wiring task for this surface.

**Auto-clear:** When a successful rotate+sync completes, `app_token_revoked` reverts to `false` → badge disappears on the next list refetch. No manual dismiss UI.

---

### Surface 3 — ProfileDrawer: Sync-now progress + completion toast (polish)

**What it is:** Polish over the existing `SyncProgressSection` + `SyncToast` components.

**SyncProgressSection (already exists):**
- Spinner: 20px circle, `--gruvax-yellow` 2px ring on `--gruvax-blue-faint` track, `animation: spin 1s linear infinite`
- Label: "Syncing…" (Space Grotesk 14px `--gruvax-text-secondary`)
- Elapsed counter: optional — shown as "(Ns)" in DM Mono 14px `--gruvax-text-muted`, updated each poll tick (approx every 1.5s)
- `aria-live="polite"` + `aria-busy="true"` on the progress container

**SyncToast (already exists):**
- Position: fixed top-right, `z-index: var(--gruvax-z-admin)` (50)
- Background: `--gruvax-success`, text: `--gruvax-white`, Barlow Condensed 16px 700
- Entry: slide-in from right (`translateX(100%) → translateX(0)`) over `--gruvax-duration-slow` (400ms) ease-decelerate
- Exit: fade out (`opacity: 1 → 0`) over `--gruvax-duration-base` (250ms) ease-standard
- Auto-dismiss: 4000ms total; exit animation begins at 3850ms
- `role="status"` + `aria-live="polite"`

P4's polish task: confirm elapsed-seconds counter renders in the existing `SyncProgressSection.sync-progress-count` slot. No new CSS classes required.

---

### Surface 4 — `/admin/settings`: sync cadence control (new control in existing SETTINGS section)

**What it is:** A new labeled select control appended to the existing GENERAL SETTINGS section (or a new "SYNC" subsection if Settings.tsx is restructured).

**Control spec:**
- Label: "SYNC CADENCE" (Barlow Condensed 16px 700, ALL CAPS, `--gruvax-blue`)
- Control: `<select>` styled to match existing settings inputs
  - Width: `min(200px, 100%)` — not full-width on desktop
  - Options: "Every 24 hours" / "Every 12 hours" / "Every 6 hours" / "Off"
  - Values: `"24h"` / `"12h"` / `"6h"` / `"off"`
  - Default displayed value: whichever is stored in `sync.cadence` setting (fetched on mount)
- Sub-label (below select): "Syncs run at 03:00, 15:00 (12h), 09:00/21:00 (6h) server time." (Space Grotesk 14px `--gruvax-text-muted` — subordinate via muted color, not a smaller size; keeps the type scale at four sizes)
- Save behavior: auto-save on `onChange` (matches existing Settings pattern) with a transient "Saved" confirmation (Space Grotesk 14px `--gruvax-success`, fades after 2s)
- Min-height: 44px on the select element

**Styling:** reuse existing `settings-select` class pattern from admin.css (or create `.settings-select` if not yet present — same border, border-radius, font, padding as other inputs on the page).

---

### Surface 5 — Kiosk: re-auth inline banner (new conditional banner below existing StalenessBar)

**What it is:** A non-blocking inline banner rendered in `KioskView` when `GET /api/session` returns `needs_reauth: true` for the bound profile. Appears directly below the existing `StalenessBar` (or in its place if `StalenessBar` is hidden — they can stack).

**Visual spec:**
- Background: `--gruvax-yellow`
- Text: `--gruvax-blue-darker`
- Font: Space Grotesk 18px 400, line-height 1.5 — matches `staleness-bar` exactly
- Padding: `var(--gruvax-space-3)` (12px) vertical × `var(--gruvax-space-4)` (16px) horizontal
- Full-width, `border-radius: 0` (flush edge-to-edge, system-bar style)
- Icon: inline SVG `KeyRound` or `AlertCircle` (18×18, `aria-hidden="true"`, Lucide-pattern, `stroke="currentColor"`)
- Mount animation: `staleness-bar-enter` keyframe (already defined in `StalenessBar.css`) — share or copy the rule
- `role="alert"` + `aria-live="polite"`
- Not dismissible — persistent while `needs_reauth` is true; clears automatically on next session refresh after a successful rotate

**Copy:** "Shelf data may be outdated — ask the owner to update the connection." (Sentence case, plain language, em dash, no technical jargon. Note: do NOT say "PAT", "token", or "API key" in user-facing copy.)

**Stacking with StalenessBar:** if both `staleness-bar` and `reauth-banner` are visible simultaneously, stack them vertically (banner above staleness bar, or vice versa — planner's discretion as long as both render). Never merge them into one element.

## Interaction States

### Diagnostics cards

| State | Treatment |
|-------|-----------|
| Loading (initial) | `.diag-skeleton` shimmer block per card |
| Loaded | Card grid renders; TanStack Query refetches every 30s |
| Refetching (background) | No visual change (background refetch — do not show a loading spinner) |
| Load error | Error banner with "Try again" button (existing `.diag-load-error` pattern) |

### Sync-now button (in ProfileDrawer)

| State | Treatment |
|-------|-----------|
| Idle | "SYNC NOW" primary button |
| In-progress | `SyncProgressSection` replaces button; spinner + "Syncing…" + elapsed |
| Complete | `SyncToast` appears (4s auto-dismiss); drawer reverts to idle state |
| Failed | Toast with error copy: "Sync failed — {error_tag}" (Space Grotesk 14px muted, or use error-colored toast variant) |

### Re-auth kiosk banner

| State | Treatment |
|-------|-----------|
| `needs_reauth: false` | Banner hidden (component returns null) |
| `needs_reauth: true` | Banner mounts with `staleness-bar-enter` animation |
| After successful rotate | `GET /api/session` returns `needs_reauth: false`; banner unmounts by returning null |

### Cadence select

| State | Treatment |
|-------|-----------|
| Loading settings | Select shows placeholder or current value from API |
| Change | `onChange` fires auto-save; optimistic UI (update local state immediately) |
| Saved | "Saved" confirmation fades in then out over 2s |
| Error | "Could not save. Try again." (Space Grotesk 14px `--gruvax-error`) |

## Animation Contract

All animations consume existing tokens — no new easing functions.

| Animation | Token | Duration | Easing |
|-----------|-------|----------|--------|
| Kiosk banner mount | `staleness-bar-enter` keyframe (existing) | `--gruvax-duration-base` (250ms) | `--gruvax-ease-decelerate` |
| Toast slide-in | `translateX(100%) → translateX(0)` | `--gruvax-duration-slow` (400ms) | `--gruvax-ease-decelerate` |
| Toast fade-out | `opacity: 1 → 0` | `--gruvax-duration-base` (250ms) | `--gruvax-ease-standard` |
| Spinner rotation | `rotate(360deg)` continuous | 1s | `linear infinite` |
| Skeleton shimmer | existing `diag-shimmer` keyframe | 1.2s | `linear infinite` |
| Cadence "Saved" fade | `opacity: 1 → 0` | `--gruvax-duration-slow` (400ms) after 1.6s delay | `--gruvax-ease-standard` |
