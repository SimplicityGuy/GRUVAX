<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server SSE bus → kiosk EventSource | device lifecycle events arrive over the per-profile channel |
| in-flight API response → kiosk | a 403 device_revoked may race the SSE event |
| old profile channel → new profile channel | reassign must fully tear down the old SSE before binding the new |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Information Disclosure | stale SSE connection after revoke/reassign | mitigate | clearBoundProfile() nulls boundProfileId → the effect cleanup closes the old EventSource; no second channel opened manually (D-07). |
| T-06-06 | Spoofing/Repudiation | 403 and SSE revoke race | mitigate | Both route through one idempotent handleTerminalRevoke (ref guard) so the kiosk exits exactly once regardless of which arrives first (D-06). |
| T-06-07 | Information Disclosure | reassign banner leaking wrong profile name | mitigate | display_name read only from the authoritative GET /api/session re-fetch for the now-bound profile (D-08, D-09) — never from the event payload (which carries only device_id). |
| T-06-SC | Tampering | npm installs | accept | No new frontend dependencies — STATE pins react-qr-code for Phase 8 only; this plan adds none. |
</threat_model>
