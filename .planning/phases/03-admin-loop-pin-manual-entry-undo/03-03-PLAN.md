<objective>
Deliver the kiosk-reveal vertical slice: every cube on the kiosk grid gains a fill-level bar, and tapping any cube opens a public side panel showing the cube's first/last boundary records, a representative sample of its contents, and a total count + fill percentage. Backed by the existing `GET /api/cubes/{unit}/{row}/{col}` endpoint extended (no new endpoint) to compute fill level and sample records from the in-memory collection snapshot (no DB during compute).

Purpose: Satisfies ROADMAP criterion 4 (CUBE-07 fill indicator + CUBE-09 reverse-lookup). This slice is PUBLIC (D-15 — visiting friends can browse). It depends on the plan-01 boundary-math helpers AND on the plan-02 admin store slice: the D-16 "EDIT THIS CUBE" shortcut in CubeContentsPanel reads `isLoggedIn` from the admin store slice defined in plan 02. Because of that store import, this plan depends on plan 02 and runs in Wave 3 (alongside plan 04, with which it shares no files). To keep zero file-ownership overlap with the admin slice, this plan owns its frontend types in a dedicated `frontend/src/api/cubeTypes.ts` module rather than the shared `types.ts` (which plan 02/04 own).

Output: Extended `units.py` endpoint, `cubeTypes.ts`, `FillBar.tsx`, `CubeContentsPanel.tsx`, fill bars wired into the kiosk grid, and the cube-tap interaction in `KioskView`.
</objective>
