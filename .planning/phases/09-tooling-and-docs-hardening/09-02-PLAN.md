<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GHCR | The build job authenticates with GITHUB_TOKEN and pushes an image; over-scoped tokens or moving action tags are supply-chain risks |
| third-party Action `uses:` → workflow runtime | A moving tag (e.g. `@v4`) can be repointed to malicious code; SHAs are immutable |
| workflow YAML → GitHub Secrets | Hardcoded credentials in YAML would leak via the public repo |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-9-PERM | Elevation of Privilege | `build.yml` permissions | mitigate | Workflow-level `permissions: contents: read`; `packages: write` scoped to the build job only; `security-events: write` scoped to the security job only. Verified by grep in Task 2 acceptance. |
| T-9-SC | Tampering | third-party Action `uses:` lines | mitigate | All `uses:` pinned to 40-char commit SHAs (verified SHAs from RESEARCH.md). Acceptance gate rejects any `@vN` tag. |
| T-9-TOKEN | Information Disclosure | `GITHUB_TOKEN` in build/cleanup | mitigate | Use the auto-provisioned `secrets.GITHUB_TOKEN` (never a PAT in plaintext); login-action only on non-PR events. |
| T-9-CSV | Information Disclosure | test.yml DB seed | mitigate | Seed ONLY `fixtures/synth_collection.sql`; CI never touches the real collection CSV or `background/` (repo-hygiene constraint). |
| T-9-SECRETSCAN | Information Disclosure | new workflow YAML files | mitigate | security.yml runs TruffleHog over the repo; no secrets are embedded in any YAML (referenced via `secrets.*`). |
</threat_model>
