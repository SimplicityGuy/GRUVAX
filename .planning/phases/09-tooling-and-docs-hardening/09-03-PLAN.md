<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dependabot PR → repo | An automated dependency bump could introduce a compromised package version |
| update-project.sh → uv/npm registries | The updater pulls new dependency versions; a malicious upstream release could land in the lockfile |
| pre-commit hook repos → developer machine | Hook code runs locally on every commit; a repointed (unfrozen) hook rev could execute arbitrary code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-9-DEPBOT | Tampering | dependabot auto-PRs | accept | dependabot opens PRs only; merge requires human review (no auto-merge configured). Low risk for a home-LAN, no-public-exposure project. |
| T-9-HOOKPIN | Tampering | `.pre-commit-config.yaml` hook revs | mitigate | Every `rev:` frozen to a 40-char SHA (verified from RESEARCH.md). Acceptance gate rejects floating tags. `pre-commit autoupdate` (in update-project.sh) re-pins to new SHAs under human review. |
| T-9-UPDLOCK | Tampering | update-project.sh dependency refresh | accept | Script ends with "Review git diff for surprising version jumps"; the operator reviews the lockfile diff before committing. |
| T-9-BANDIT | Information Disclosure | Python source (SQL injection risk) | mitigate | bandit hook stays in pre-commit (`-x tests -s B608`) — preserves the Phase 8 static-analysis control. |
</threat_model>
