<objective>
Close the single UAT gap (05-UAT.md test 6, severity major): SEG-05 label-contiguity is
NOT enforced where edits actually happen. The rebuilt admin editor commits cut/insert edits
DIRECTLY via `PUT /cut` (`put_bin_cut`) and `POST /insert-cut` (`insert_cut`). `validate_contiguity`
is wired ONLY into `POST /cubes/validate` + the bulk path (cubes.py:468), which feed the
now-orphaned `DiffPreviewSheet` at `/admin/preview` — no live route navigates there. A
contiguity-violating cut is therefore not blocked on the live paths.

CHOSEN DIRECTION: (A) — enforce server-side on the live write paths, surface the error in the
sheet, and REMOVE the orphaned preview surface.

Rationale (state in commit + leave a code comment):
- (A) is the smallest, lowest-risk change that closes the requirement gap on the paths the
  rebuilt editor actually uses. Adding one `validate_contiguity` call (gated before the DB
  transaction) per direct write path is far less surface area than re-routing the entire editor
  through validate→preview→commit (direction B), which would re-introduce a two-step flow the
  owner explicitly accepted dropping (05-UAT.md test 5 note: "Direct-save … accepted by owner
  as-is").
- The orphaned `DiffPreviewSheet`/`/admin/preview` is dead code. The owner accepted direct-save,
  so it is REMOVED (not wired in later) to avoid a reachable-but-broken UI surface and to keep
  the route table honest. `HistoryView`'s revert/history client functions are independent and stay.

Purpose: SEG-05 must hold on the live edit paths, with a reachable plain-language UI block.
Output: 400 `{type:"contiguity_error", message}` on scatter from both direct write paths; the
RecordPickerSheet renders that message and stays open; the orphaned preview route is gone; a
regression test guards the direct path; all gates green.
</objective>
