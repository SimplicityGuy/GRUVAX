<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser network state → kiosk UI gating | `navigator.onLine` is untrustworthy on a LAN; SSE state is the real signal (PITFALLS 35) |
| client-side control gating → server | Disabling controls is UX, NOT a security boundary; server-side authz remains the control |
| SSE `device_revoked` event → terminal device state | A revoked device must reach its terminal state even while the offline banner could render |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-07 | Elevation of Privilege | Client-side gating of search / profile-switch / cube taps | accept | Gating is UX only; bypassing it (e.g. via devtools) cannot reach data the server would not already serve. Server-side authz (admin PIN, device binding from Phase 6) is the real control and is unaffected by this plan. |
| T-09-08 | Denial of Service (terminal-state masking) | `device_revoked` terminal path vs `!sseConnected` banner render | mitigate | A revoked device must NOT be hidden behind the generic offline banner. The device_revoked handler calls `triggerRevoke()` (owned by App.tsx, which navigates away from the kiosk) independent of the banner; Task 3 explicitly preserves this and the acceptance test asserts the existing device-revoke SSE test still passes (Phase 6 DEV-05). |
| T-09-09 | Spoofing | `navigator.onLine=true` masking a real server outage | mitigate | The banner renders on `!sseConnected` only; `navigator.onLine` selects copy text only (PITFALLS 35). A LAN-up/server-down state still shows the banner with the "Can't reach GRUVAX" variant. |
| T-09-10 | Tampering | Stale cache served as fresh / dismissed diff badge re-appearing on reconnect | mitigate | OFF-04 reuses the existing `server_hello` resync+invalidate; the dismissed-diff badge stays dismissed via the existing `dismissed_diff_at` vs `last_sync_at` check (PITFALLS 37) — Task 3 explicitly does not regress it, and RecentlyPulled is local Zustand/sessionStorage (never refetched). |
</threat_model>
