## Cross-Phase Integration Findings

### 🛑 Blockers

**B-01 — `collection_changed` SSE published but never consumed by the kiosk** (SYN-01, SYN-02, API-02)
`profile_sync.py:356` publishes `collection_changed` after every successful sync; `KioskView.tsx`
registers `boundary_changed` / `admin_editing` / `server_hello` / `server_shutdown` but **not**
`collection_changed`, and `resync()` invalidates only `['units']`/`['cubes']`, never `['search']`.
→ After a nightly or "Sync now" refresh the kiosk shows pre-sync results until a manual reload.

**B-02 — `profile_id` omitted when `boundProfileId` is null → search/locate 422** (API-02, DEV-02)
`client.ts:19,30` only sends `profile_id` when truthy; `search.py:43` / `locate.py:78` declare
`profile_id: str = Query()` with no default. Search fires gated only on query length, not on
`boundProfileId`. → If a user types before session bootstrap resolves the bound profile (multi-profile
or device-paired deployments), the backend returns 422 and the results list errors. Low risk in a
steady-state single-profile deploy; real for the multi-user flows this milestone exists to deliver.

### ⚠️ Warnings (tech debt — not flow-breaking)

- **W-01** Kiosk `StalenessBar` shows the default profile's staleness, not the bound profile's (SYN-02).
- **W-02** `PairView` navigates to `/` without refreshing the session store → SSE may lag ≤5 min after a fresh pair (DEV-02, DEV-03).
- **W-03** Admin `boundary_changed` fires on the default profile's bus only → non-default kiosks miss live boundary edits (API-02). *Explicitly deferred as D2-04 in `app.py:280`.*
- **W-04** No kiosk handler for `device_revoked`/`device_reassigned` → revoked kiosk reverts to picker only on its next request, not promptly (DEV-01, DEV-02).
