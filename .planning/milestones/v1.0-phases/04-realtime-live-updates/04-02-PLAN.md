<objective>
Make the visitor's highlight follow a relocated record. When a `boundary_changed` SSE event arrives and the visitor has an active selection, the kiosk re-runs `locate` for that record so the highlight relocates to the record's new cube, presented as a re-glow (old cube fades off, new cube springs on). This is the user-visible payoff of the spine: the kiosk visibly tracks where a record moved to.

Purpose: Satisfies D-05 (highlight follows the record) and D-06 (re-glow, not slide) — an extension beyond ARCHITECTURE's consumer sketch which only invalidated cube keys. After this plan, a visitor watching a selected record sees its highlight jump to the new cube the instant the owner re-files it.
Output: The `boundary_changed` handler in KioskView re-locates the active selection via the existing imperative `locateRelease` → `setLocateResult` path (which bumps `animationToken`, firing the existing GSAP re-glow timeline — no new animation code), plus the frontend EventSource consumer test scaffold.
</objective>
