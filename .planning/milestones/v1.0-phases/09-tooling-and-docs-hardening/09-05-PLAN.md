<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| compose.override.yaml → prod host | If the dev-only override reaches the prod host, `docker compose up` auto-builds from source instead of pulling the published image (deploy integrity failure) |
| GHCR image → prod runtime | The prod host pulls and runs `ghcr.io/simplicityguy/gruvax:latest`; a tampered or unauthenticated image would run untrusted code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-9-OVERRIDE | Tampering | compose.override.yaml on prod | mitigate | Override is gitignored (verified via `git check-ignore`); a committed `.example` documents it is dev-only; the runbook explicitly states prod must not have it (Pitfall 3). |
| T-9-IMGSPOOF | Spoofing | GHCR image pull | accept | GHCR public package for an open-source, home-LAN, no-public-exposure project; image is published only by the SHA-pinned, least-privilege build job (Plan 02). Accepted per RESEARCH.md Security Domain. |
| T-9-LUXLEAK | Information Disclosure | host names in compose/runbook | mitigate | Genericize all lux references to neutral placeholders (D-10); acceptance gates assert `grep -c lux` returns 0 in both files. No new runtime config introduced. |
</threat_model>
