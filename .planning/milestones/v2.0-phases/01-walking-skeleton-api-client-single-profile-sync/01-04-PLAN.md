<!-- Endpoint contract -->

POST /api/admin/profiles/{profile_id}/sync
  Auth: Depends(require_admin) — session cookie + CSRF double-submit (existing v1 dependency)
  Path: profile_id (UUID string; validated)
  Body: empty (or optional {} — Pydantic model with extra="forbid")
  200: {"status":"ok", "item_count": int, "took_ms": float, "user_id": str}
  401: missing/invalid session cookie
  403: missing CSRF
  404: profile_id not found OR soft-deleted
  409: {"type":"already_in_progress"} — SyncInProgress raised
  400: invalid UUID path param
  401-like: {"type":"pat_rejected"} on PATRejected
  503: {"type":"rate_limited_upstream"} on RateLimitExhausted; {"type":"upstream_unavailable"} on ServerError/NetworkError

CRITICAL — handler MUST NOT use Depends(get_pool):
  The standard admin-handler analog (editing.py) uses `pool: Any = Depends(get_pool)`
  which keeps the pool slot attached for the request lifetime. For sync_profile (~tens of
  seconds), this defeats Plan 03's Pitfall-6 mitigation: a concurrent admin request would
  block waiting for the pool slot the sync is holding.

  RIGHT pattern for this handler:
    db_pool = request.app.state.db_pool
    async with db_pool.connection() as conn, conn.cursor() as cur:
        await cur.execute("SELECT 1 FROM gruvax.profiles WHERE id = %s::uuid AND deleted_at IS NULL", (str(uid),))
        if await cur.fetchone() is None:
            raise HTTPException(404, ...)
    # async-with block CLOSED here — pool slot released BEFORE awaiting sync_profile
    result = await sync_profile(str(uid), request.app.state)

<!-- CLI contracts -->

gruvax-set-pat --profile <display_name>
  stdin: PAT plaintext (either piped or interactive getpass when TTY)
  No --pat flag; no env var fallback (D-07)
  Flow per D-08/D-09:
    1. Validate PAT prefix dscg_ + length >= 50; else sys.exit non-zero.
    2. Construct DiscogsographyClient(base_url=settings.DISCOGSOGRAPHY_BASE_URL, pat=pat).
    3. await client.first_page() — captures envelope. On PATRejected: sys.exit with "PAT rejected" message; DB unchanged.
    4. Validate envelope["releases"][0] has "catalog_number" key — if missing, sys.exit "discogsography sample release missing catalog_number — refusing".
    5. Capture new_user_id = str(envelope["user_id"]).
    6. SELECT discogsography_user_id, deleted_at FROM gruvax.profiles WHERE LOWER(display_name) = LOWER(:profile) AND deleted_at IS NULL.
       If no row → sys.exit f"No profile named {profile!r}".
       If existing discogsography_user_id IS NOT NULL AND != new_user_id → sys.exit "PAT belongs to a different discogsography user (was <old>, got <new>). Soft-delete the profile first if you really intend to switch." (D-09 verbatim).
    7. UPDATE gruvax.profiles SET app_token_encrypted=encrypt_pat(pat), app_token_revoked=FALSE,
         discogsography_user_id=COALESCE(discogsography_user_id, :new_user_id::uuid)
         WHERE LOWER(display_name)=LOWER(:profile) AND deleted_at IS NULL.
    8. Print success message: f"PAT stored for profile {profile!r} (user_id={new_user_id}). Run `gruvax-sync --profile {profile}` to perform the full sync."
  Failure: sys.exit "<plain English>"; no exception traceback.

gruvax-sync --profile <display_name>
  PIN input branch (TTY-aware):
    pin = getpass.getpass("Enter admin PIN (4 digits): ") if sys.stdin.isatty() else sys.stdin.readline().rstrip("\n")
  Validate: pin.isdigit() and len(pin) == 4 — else sys.exit "PIN must be 4 numeric digits".
  Flow per PATTERNS §2 + RESEARCH §Open Q1 RESOLVED:
    1. POST {base_url}/api/admin/login with {"pin": pin}; capture session cookie + CSRF from response.
       If non-200 → sys.exit f"Admin login failed: HTTP {status}".
    2. Resolve profile_id from display_name via a thin DB lookup (the CLI talks to BOTH the API and the DB in the single-deployment model).
    3. POST /api/admin/profiles/{profile_id}/sync with cookies + X-CSRF-Token header; httpx.Timeout(read=120.0).
    4. On 200: print response JSON; exit 0.
    5. On non-200: print response body to stderr; sys.exit non-zero with message including status_code.
